## Repo-Oriented Usage

For this codebase, the practical flow is:
1. Classify the mission in this document first.
2. If it is `HORIZONS`, use HORIZONS IDs to fetch ephemerides and generate `*-cheb.json` artifacts.
3. If it is `NAIF/SPICE-only`, plan a kernel-based ingestion path (not covered by current HORIZONS-first scripts).
4. If it is `partial/booster-only` or `unknown/unverified`, do not treat it as full primary-spacecraft coverage.

Current repo tooling and metadata are primarily HORIZONS-oriented (for example `scripts/fetch-mission-blurbs.py` and `docs/horizons-blurbs/`).

Current mission set implemented in the app is:
- `chandrayaan3`
- `chandrayaan2`
- `artemis1`
- `apollo11-sivb`
- `apollo10-lm`

### `docs/horizons-blurbs/` generation policy

- `docs/horizons-blurbs/mission-index.json` is generated by `scripts/fetch-mission-blurbs.py`.
- The `raw/`, `markdown/`, `metadata/`, and `mission-index.json` files should be treated as a synchronized generated set.
- If any mission blurb data is refreshed, regenerate the full set and commit the index update together.

## When to Use HORIZONS vs NAIF/SPICE

| System | Use it when | Strengths | Limitations in this repo context |
|--------|-------------|-----------|----------------------------------|
| **JPL HORIZONS** | Mission/object has a confirmed HORIZONS spacecraft ID | Fast HTTP API, easy lookup, fits existing repo workflow | Coverage is uneven by mission/agency; some entries are only stages/boosters |
| **NAIF/SPICE** | Mission is missing from HORIZONS but has NAIF kernels | Mission-native kernels, richer geometry/frame support | Requires kernel management + SPICE tooling; no turnkey SPICE ingestion pipeline here yet |

---

## Availability Classes

| Class | Meaning |
|------|---------|
| **HORIZONS** | Primary mission object (or mission-relevant object) confirmed in HORIZONS |
| **NAIF/SPICE-only** | Available via NAIF kernels, not available in HORIZONS API for current repo workflow |
| **Partial/booster-only** | Only stage/booster/debris object is known, not primary spacecraft coverage |
| **Unknown/unverified** | No confirmed HORIZONS ID or NAIF coverage verification in this document |

## Access Methods and References

### HORIZONS access

- Web interface: https://ssd.jpl.nasa.gov/horizons/app.html
- API example:
  ```
  https://ssd.jpl.nasa.gov/api/horizons.api?format=text&COMMAND='-85'
  ```
- Lookup API example:
  ```
  https://ssd.jpl.nasa.gov/api/horizons_lookup.api?sstr=LRO&group=sct
  ```

### NAIF/SPICE access

- NAIF lunar data index: [NAIF Lunar Data](https://naif.jpl.nasa.gov/naif/data_lunar.html)
- NAIF ID reference: [NAIF ID Codes](https://naif.jpl.nasa.gov/pub/naif/toolkit_docs/C/req/naif_ids.html)

### ID notes

- Spacecraft IDs in HORIZONS are negative integers.
- Apollo S-IVB objects use the `-399XXX` pattern.
- Stage/booster IDs should not be treated as equivalent to primary spacecraft availability.

### General references

- [JPL HORIZONS System](https://ssd.jpl.nasa.gov/horizons/)
